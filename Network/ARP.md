
# 基本原理

ARP 也称为地址解析协议. 

首先在数据链路层, 如果要实现通信, 我们要知道对方的 mac 地址. 对方要与我通信, 也需要知道我的 mac 地址. 我们用泛洪的方式来通告自己的 mac 地址. 也就说此时网络中的所有设备都在一个广播域里, 大家都能收到别人发出的 mac 地址通告. 这个通告也称为免费 arp 广播. 

当我要发送信息的时候, 我们会对数据进行而成封装 `dmac | smac | data`, 在广播网络中的所有主机都会收到这个帧, 这些设备发现目的 mac 地址不是自己时, 就会丢弃这个帧. 目标主机发现这个帧的目的 mac 地址是自己时, 才会接受并解封装, 并知道这个帧是以 smac 为 mac 地址的设备发过来的. 

如果没有交换机, 那这种来回的通信都需要通过广播泛洪来实现了. 网络中就充斥着中大量的报文. 因为不是目的地址的设备, 也会收到泛洪来的报文, 占用了带宽和资源. 

交换机解决了什么问题? 有了交换机后, 因为所有的设备都连上了交换机的接口. 在大家第一发送 ARP 泛洪报文后, 交换机的接口接受到帧的源 mac 地址. 换句话说, 交换机知道接口 1 下有设备 A, 接口 2 下有设备 B. 当设备 A 要向设备 B 发送报文时, 交换机会直接把数据从接口 2发送出去, 而不会把数据丢给其他接口了. 这样虽然第一次的免费 arp 报文少不了. 但是一旦 mac 地址表收敛 (mac 地址和接口的对应关系稳定了), 那么后面数据之间的转发就可以认为是点到点的传输. 

基本的功能实现了, 我们看看实际中还有些什么问题
1. mac 地址老化的问题? 
	现实中我的网络并非一成不变的, 老的设备走了, 新的设备进来. 要删除老的 mac 地址, 添加新的 mac 地址, 就需要没过一段时间去做一次免费的 arp 广播. 这个会造成几个问题
	1. 广播报文的扩散问题. 从安全的角度看, 我希望做部门之间的隔离. 两个部门的终端接到同一台交换机之后, 他们会各自收到对方发来的免费广播, 知道对方的 mac 地址. 那么他们之间就可以通信. 
	2. 从成本的角度看, 为了使两个部门互相不能通信, 我们就需要买两台交换机. 
	为了解决上面两个问题, 我们引入 [[VLAN]]这个技术
	



### 区分 mac 表和 ARP 表 
在终端设备和交换机上都有 ARP 表
* ARP 表是 mac 地址和 IP 地址的映射
* MAC 表是 mac 地址和接口的映射

# ARP 的报文

### 免费 ARP
当设备修改和新设置 ip 地址的时候, 就会发送免费 ARP, 用来检测网络中是否有 IP 地址冲突

### ARP Request
当知道对方的 ip 地址, 但是不知道对方的 mac 地址的时候, 就会发送 ARP request
请求的帧头部 源 mac 地址是自己网卡的 mac 地址, 目标 mac 地址是广播 mac 地址. 
1. 当交换机收到 ARP 请求时, 会学习帧头的源 mac 地址. 然后进行泛洪
2. 对端主机收到 arp 请求后, 发现询问的是自己的 mac 地址, 会发送 ARP Reply (单播). 其中目的 mac 地址, 是收到的帧的源 mac 地址, 源 mac 地址是自己的接口 mac 地址.

### ARP 表老化时间
一般主机的 ARP 表老化时间是 10-20 分钟, 比如 win10

### ARP Proxy 代理

当收到一个 ARP 请求报文时, 发现其请求的 IP 地址不是自己, 则会进行如下操作
* 查询自己的路由, 如果有到目的地址的路由, 则把自己的 mac 地址发送给请求方
